Failure Modes and Remediations in LLM-Driven Automated Machine Learning: A Case Study on Network Intrusion Detection

The setup is simple to describe. Take Karpathy's autoresearch idea, where an LLM sits in a closed loop rewriting its own experiment code and reacting to the scores, and point it at tabular classification: binary network intrusion detection on CICIDS-2017, about 2.8 million labeled flow records. Claude Sonnet 4.6 rewrites a detector script each iteration; the script runs as a subprocess against a 300-second training budget; a fixed evaluator the agent is never allowed to touch scores it against a held-out test set. A separate Haiku instance compresses the experiment ledger into 400 tokens after each run so the history fits in context. I ran four sessions, 27 iterations in all, on a consumer CPU with no GPU. Total API cost for the whole study: about $3.15.

CICIDS-2017 was a deliberate choice because it's close to solved. Tree ensembles sit above 0.999 weighted F1 in the published literature, and my loop got there almost immediately: the first successful run of Session 1 hit val F1 = 0.9991, and nothing in the following 26 runs improved on it by more than 0.0002. The best model, a soft-voting ensemble of HistGradientBoosting, RandomForest, and ExtraTrees, finished at test F1 = 0.9991, consistent with the benchmarks. If the question had been "can an LLM do AutoML on a solved problem," the paper would be one paragraph long. The interesting material is what went wrong across those 27 runs, because when model quality is essentially guaranteed, every remaining failure is a systems failure, and you get an unusually clean look at them.

The paper documents six, each reproduced across runs, root-caused, and fixed:

  1. Revert-loop stagnation. When a run fails to improve, the orchestrator reverts to the best detector, which means the next prompt is identical to the last one, which means the model produces the same code. Sonnet regenerated byte-for-byte identical detectors six runs straight in Session 1. The fix is to inject the reverted attempt into the next prompt with a note saying "this was reverted, don't write it again."
  2. Feature engineering distribution mismatch. Left unconstrained, Sonnet invented 58 to 67 engineered features per run. They were applied to train and validation, but the fixed evaluator only has the raw test columns, and the missing features get silently zero-filled. Validation looked perfect while test-set recall on attacks collapsed from 0.999 to 0.46 in the worst run. The fix was prohibiting feature engineering in the prompt, and, notably, explaining the causal mechanism rather than just issuing the rule.
  3. Resource budget violations. Code that is entirely correct and also wants more time than the 360-second kill limit allows, like a stacking classifier running 3-fold cross-validation over two million rows. The fix was injecting empirically measured per-algorithm runtimes and a mandatory timing probe on a 10% subsample. It worked for ensembles and failed for neural networks, whose cost scales with epochs, not samples.
  4. Platform encoding crashes. Sonnet put a Unicode arrow in a log statement. Windows stdout defaults to cp1252, which can't represent it, and the subprocess died before any training happened.
  5. Deprecated API incompatibilities. The model writes against libraries as it remembers them from training data, not as they exist in the deployed environment: a removed sklearn parameter, a pickle module that can no longer serialize PyTorch tensors. This one never fully goes away; the "known broken APIs" list in the prompt requires maintenance as environments evolve.
  6. Cross-process pickle scope errors. The agent defined a custom ensemble class in its script and serialized an instance. Pickle recorded it as living in __main__, and the separate evaluator process, whose __main__ contains no such class, couldn't load it. The fix imports the detector's class definitions into the evaluator's namespace before unpickling.

What strikes me about that list is how little of it is machine learning. One entry is a genuine methodological trap, the distribution mismatch. Everything else is a failure class any engineer who has operated software in production would recognize: environment drift, resource limits, serialization, encodings. The model in the loop was a competent data scientist and an unreliable software engineer, and the orchestration layer had to make up the difference.

The remediations are also measurably what drove progress. Session 1 produced 2 evaluable runs out of 10. Session 4, with all the fixes in place, produced 4 confirmed results out of 7 with no stagnation and no validation/test gap. Same model the whole time. That's the central finding: near a performance ceiling, the primary challenge of LLM-driven AutoML is not model quality but system reliability. The ceiling itself creates a second-order problem worth knowing about, because at 0.999 the improvement signal that decides whether to keep a model is smaller than the run-to-run noise from RandomForest thread scheduling, so the loop can't even tell progress from luck.

One behavioral observation that didn't fit the taxonomy but stayed with me: despite explicit instructions to explore diverse architectures, Sonnet kept returning to the same HGB + RF + ExtraTrees ensemble across all four sessions. Its implicit preference for the near-optimal answer overrode the diversity instructions every time. The only two non-tree attempts in 27 runs were PyTorch MLPs, and both timed out. If you want an agent to explore, you apparently have to build the curriculum into the harness; asking nicely does not work.

The framework is open source, dataset-agnostic, and introspects whatever tabular dataset you point it at to build its own prompt context.
